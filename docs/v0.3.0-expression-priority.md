# Clawpet v0.3.0 Spec — Expression Priority and Concurrency Groundwork

## Goal

Make **OpenClaw expression** feel intentional instead of disposable.

In `v0.3.0`, Clawpet should stop treating all emits like interchangeable ambient telemetry. Contextual OpenClaw expression should linger long enough to be perceived, lower-priority system-signal events should not instantly stomp it, and the activity log should clearly explain what was shown, suppressed, or replaced.

This release also lays the groundwork for future multi-conversation / multi-session concurrency without trying to solve a full multi-slot avatar UI yet.

---

## Product direction

Clawpet should behave less like **latest event wins** and more like:

> **the most meaningful active event wins**

That means:
- **system signal** remains fast, ambient, and replaceable
- **OpenClaw expression** becomes sticky for a short window
- **user-requested** emits remain the strongest manual/intended surface
- the **activity log** becomes the truth surface for all events, including suppressed ones

---

## In scope for v0.3.0

### Must-have
- OpenClaw expression linger/hold behavior
- explicit emit priority rules
- suppression/replacement logic
- cooldown / duplicate suppression
- activity-log visibility into arbitration outcomes
- event metadata groundwork for later multi-session concurrency

### Out of scope
- new avatar frames
- richer completion/decay animation systems
- user-facing behavior tuning controls
- full multi-slot / multi-avatar UI
- built-in critical-alert framework

---

## Event classes

For this release, Clawpet should reason about three user-visible event classes:

1. **user-requested**
2. **OpenClaw expression**
3. **system signal**

### Meanings

#### 1) user-requested
Explicit manual emits or routines Nick intentionally asked for.

Examples:
- manual `clawpet send ...`
- explicit celebrations / reminders / configured reactions the user asked for
- a user-configured “show `alert` when heartbeat completes” behavior

#### 2) OpenClaw expression
Optional contextual/model-driven commentary produced by OpenClaw.

Examples:
- a short contextual remark about progress
- a grounded reaction to ambiguity, success, friction, or completion

#### 3) system signal
Zero-token runtime/daemon operational telemetry.

Examples:
- daemon tool/session mirroring
- runtime/internal plumbing updates
- local activity pulses that are not expressive/manual avatar decisions

---

## Priority model

### Priority order

Highest to lowest:

1. **user-requested**
2. **OpenClaw expression**
3. **system signal**

### Core rule

A lower-priority event must not immediately replace a higher-priority event whose minimum linger window is still active.

### Initial replacement rules

#### user-requested
- can replace `OpenClaw expression`
- can replace `system signal`
- can be replaced only by a newer `user-requested` emit unless a later explicit policy is introduced

#### OpenClaw expression
- can replace `system signal`
- cannot be replaced by `system signal` during its linger window
- can be replaced by `user-requested`
- can be replaced by a newer `OpenClaw expression` if the newer one is materially different and allowed by cooldown rules

#### system signal
- can replace other `system signal` events freely
- cannot replace active `OpenClaw expression` during linger
- cannot replace active `user-requested` during linger

## Linger model

Linger is the practical centerpiece of this release.

Each ingested event should carry:
- `source`
- `priorityClass`
- `state`
- `message`
- `lingerMs`
- `createdAt`
- `replacePolicy`

### Initial linger defaults

#### system signal
- `lingerMs`: very short
- goal: feels ambient and responsive
- expected behavior: replaceable almost immediately by later system-signal activity

Suggested starting point:
- `1500–2500ms`

#### OpenClaw expression
- `lingerMs`: medium
- goal: readable, emotionally legible, not instantly stomped

Suggested starting point:
- `6000–12000ms`

#### user-requested
- `lingerMs`: strongest/manual
- goal: explicitly intended by the user, should feel honored

Suggested starting point:
- `8000–15000ms`

### Important rule

For `v0.3.0`, **state and message should linger together**.

Do not attempt separate state-vs-bubble decay semantics yet. That likely belongs to a later release closer to `v1.0` when avatar frames/animation depth exist to support it well.

---

## Cooldowns and duplicate suppression

Once expression lingers longer, Clawpet must avoid spam and churn.

### Required behaviors

#### Expression duplicate suppression
Do not show near-identical OpenClaw expression bubbles back-to-back unless enough time has passed or the event meaning materially changed.

Examples:
- avoid repeating the same or almost-the-same expression line repeatedly
- avoid bouncing between two tiny variants that feel identical

#### Expression cooldown
Add a minimum cooldown for new OpenClaw expression emits so contextual remarks stay occasional and meaningful.

This cooldown can be simple in `v0.3.0`:
- no user-facing tuning yet
- just a sensible default constant/policy

#### System-signal churn suppression
When repeated system-signal events arrive during an active higher-priority linger window:
- they should still be recorded in the log
- they should generally not keep trying to visually preempt foreground presentation

## Foreground avatar behavior

The avatar can only present **one foreground event at a time**.

For `v0.3.0`, keep this simple:
- one foreground slot
- one active displayed state/message
- event arbitration decides whether a new event is shown, suppressed, or replaces the current foreground event

### Foreground selection principle

The avatar should display the **highest-priority currently eligible event**.

### Example

If an OpenClaw expression is active and lingering:
- incoming daemon/runtime system-signal events should not replace it
- those events may still be recorded in the log as suppressed or skipped due to active expression hold

If a user-requested emit arrives:
- it may replace the active OpenClaw expression immediately

---

## Activity log behavior

The activity log is the authoritative truth surface.

It should record **all important ingested events**, not only the one currently shown by the avatar.

### New need in v0.3.0

The log should communicate arbitration outcomes, not just raw event content.

Each entry should be able to express outcomes such as:
- `shown`
- `replaced`
- `suppressed`
- `skipped by cooldown`
- `deduplicated`

### Why this matters

Without this, expression-persistence behavior will feel mysterious:
- the avatar won’t change
- the system will look broken or random
- users won’t know whether events were ignored, delayed, or intentionally suppressed

### Example log phrasing

Potential user-facing patterns:
- `system signal shown`
- `system signal suppressed · active OpenClaw expression`
- `OpenClaw expression shown`
- `OpenClaw expression skipped · cooldown`
- `user-requested shown · replaced OpenClaw expression`

Exact copy can change later; the important part is that the outcome is legible.

## Concurrency groundwork

This release should prepare for the real future problem:
- Nick DMing Dawn in Discord
- a team/work channel also active
- a heartbeat completing
- background tasks/subagents finishing
- multiple concurrent sources competing for avatar presence

### v0.3.0 goal

Do **not** solve full concurrency UX yet.

Instead, introduce event metadata that will make later concurrency sane.

### Recommended metadata fields

At minimum, events should be able to carry:
- `sourceClass` (`system signal`, `OpenClaw expression`, `user-requested`)
- `sourceInstance` or internal provenance if needed
- `sessionKey` / conversation identifier when available
- `channel` / surface when available
- `priority`
- `lingerMs`
- `status`
- `replacedByEventId` or `suppressedReason` when relevant

### Immediate design split

#### Avatar
- curated foreground presentation
- one active event at a time

#### Log
- complete event history
- includes shown + suppressed + replaced + skipped events

This separation should become an explicit architectural rule.

---

## Arbitration model

Internally, separate the pipeline into four conceptual stages:

1. **event ingest**
2. **event arbitration**
3. **foreground presentation**
4. **log persistence**

### Why this split matters

Later, Clawpet will need clean answers to:
- what happened?
- what got shown?
- why did this event win?
- what was suppressed?
- which conversation/session did the event belong to?

If ingest, display, and logging stay tangled together, multi-query behavior will get messy fast.

## Example scenarios

### Scenario 1: expression protected from system signal
1. OpenClaw emits `focused` with message `This one’s being slippery.`
2. expression enters foreground with `lingerMs=8000`
3. daemon emits `read`
4. daemon emits `exec`

Expected:
- avatar keeps showing the OpenClaw expression during linger
- system-signal events are logged as suppressed
- after linger expires, later system-signal events may resume foreground replacement

### Scenario 2: user-requested beats expression
1. OpenClaw expression is active
2. user-requested manual emit arrives

Expected:
- user-requested emit replaces active expression immediately
- log records that user-requested replaced OpenClaw expression

### Scenario 3: repeated expression spam avoided
1. OpenClaw emits contextual remark
2. shortly after, another near-identical contextual remark is proposed

Expected:
- second expression is skipped by duplicate/cooldown rules
- log records the skip reason

### Scenario 4: future concurrency groundwork
1. direct Discord DM activity creates an OpenClaw expression
2. a team Discord channel emits system-signal-worthy activity
3. a heartbeat completes

Expected in `v0.3.0`:
- one foreground event only
- everything still appears in the log with provenance and arbitration outcome
- architecture is ready for later session-aware selection policies

## Implementation plan

### 1) Extend runtime event model
Add internal metadata for:
- priority class
- linger duration
- arbitration outcome
- provenance/session fields where available

### 2) Add arbitration layer
Before foreground display updates:
- compare incoming event vs active foreground event
- evaluate priority + linger window + cooldown + dedupe rules
- decide: `show`, `replace`, `suppress`, or `skip`

### 3) Update foreground presentation logic
Ensure display updates flow through arbitration instead of direct latest-event replacement.

### 4) Update log model/UI
Show arbitration outcomes clearly in the activity log.

### 5) Preserve compatibility
Keep older runtime/CLI behavior working where practical.
If older payloads lack some new metadata, infer sensible defaults.

## Release checklist

- define event metadata shape for priority/linger/arbitration
- implement active-foreground hold behavior
- block system-signal replacement during active expression linger
- allow user-requested replacement over active expression
- add expression cooldown + duplicate suppression
- log shown/replaced/suppressed/skipped outcomes
- pass `npm run check`
- pass `npm run build`
- pass `npm test`
- pass `cargo check --manifest-path src-tauri/Cargo.toml`
- verify live behavior on paired runtime
- validate a mixed-event scenario with expression + system-signal overlap

## Release framing

`v0.3.0` should be the release where Clawpet gains a real **foreground behavior model**.

`v0.2.0` clarified the surfaces.
`v0.3.0` should clarify **who gets to speak, for how long, and why**.
