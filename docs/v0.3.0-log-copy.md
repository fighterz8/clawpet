# Clawpals v0.3.0 Activity Log Wording

These are the recommended user-facing wording patterns for arbitration-aware log entries.

## Outcome labels

Use one short outcome label per row:
- `shown`
- `replaced`
- `suppressed`
- `skipped`

## Source labels

Keep the current user-facing classes:
- `system signal`
- `OpenClaw expression`
- `user-requested`

## Preferred row pattern

`<state> · <source> · <outcome>`

Examples:
- `focused · OpenClaw expression · shown`
- `thinking · system signal · suppressed`
- `happy · user-requested · replaced`

## Reason copy

### Suppressed
- `suppressed · active OpenClaw expression`
- `suppressed · active user-requested emit`

### Replaced
- `replaced · newer OpenClaw expression`
- `replaced · user-requested override`
- `replaced · newer system signal`

### Skipped
- `skipped · cooldown`
- `skipped · duplicate expression`

## Tone rules
- Short, factual, readable
- No internal jargon like “arbitration engine” in the visible UI
- Avoid implying failure when suppression is intentional
- Keep the log legible at a glance before drilling into implementation detail

## Recommended UI mapping

### Main message column
Keep what the avatar tried to say or do:
- bubble text if present
- otherwise message text
- otherwise `state → <state>`

### Meta column
Show compact truth:
- `<state> · <source> · <outcome>`
- if there is a reason, append a small suffix or secondary note

Example:
- `focused · system signal · suppressed`
- secondary note: `active OpenClaw expression`

## Empty-log helper copy

Recommended updated empty state:

`No events yet. This log records system signal, OpenClaw expression, and user-requested emits, including events that were shown, suppressed, replaced, or skipped.`
